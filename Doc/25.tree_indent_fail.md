## 문제 분석

### 핵심 문제
1. **WPF 리소스 검색 순서**: ControlTemplate 내부의 `StaticResource` 참조가 예상과 다른 리소스를 찾을 수 있음
2. **Template 상속**: `BasedOn`을 사용하면 원본 Template이 간섭할 수 있음
3. **스타일 우선순위**: 여러 스타일이 적용될 때 예상과 다른 스타일이 선택될 수 있음

- `ItemsPresenter`의 `Margin="6,0,0,0"`이 명시적으로 설정되어 있음
- `BasedOn`을 제거하여 다른 스타일의 간섭을 방지함
- Template을 완전히 오버라이드함

### 실제 문제
- 런타임에서 실제로 적용되는 값이 다를 수 있음
- WPF의 내부 렌더링 로직이 예상과 다를 수 있음
- 다른 스타일이나 리소스가 나중에 적용될 수 있음

## 참고 자료

### WPF TreeView Indent 제어 방법
1. **ItemsPresenter Margin**: 각 레벨의 들여쓰기 (정석)
2. **TreeViewItem Margin**: 노드 간 간격 (들여쓰기 아님)
3. **TreeViewItem Padding**: 콘텐츠 내부 여백 (들여쓰기 아님)

### 리소스 검색 순서
1. 현재 요소의 리소스
2. 부모 요소의 리소스
3. 애플리케이션 리소스 (`App.xaml`)
4. 테마 리소스 (`VSCodeTheme.xaml`)
5. 시스템 리소스

### StaticResource vs DynamicResource
- **StaticResource**: 컴파일 타임에 리소스 해석, 성능 좋음, 오버라이드 어려움
- **DynamicResource**: 런타임에 리소스 해석, 성능 약간 느림, 오버라이드 쉬움

# ✅ 결론 먼저 (중요)

> **현재 보이는 8px 들여쓰기는 `ItemsPresenter.Margin`이 아니다.**
> 그래서 Margin을 6으로 바꿔도 **아무 변화가 없다.**

👉 실제 indent의 정체는 **TreeViewItem 내부의 “Indent Column Width”** 이다.

# 🔥 실제 원인 (핵심)

당신이 쓰는 `VSCodeTreeViewItemStyle` 계열 템플릿은
**표준 TreeViewItem 구조가 아니다.**

대부분 VSCode 스타일 TreeView는 내부가 이렇게 되어 있음 ⬇️

```text
Grid
 ├─ Column 0 : Indent / Glyph 영역 (← 여기가 8px의 근원)
 ├─ Column 1 : ToggleButton
 ├─ Column 2 : Header(Content)
```

즉,

* ❌ `ItemsPresenter.Margin` → **자식 노드 시작 위치**
* ❌ `TreeViewItem.Margin / Padding`
* ❌ 리소스 `VSCodeTreeIndent`

이 아니라,

> ✅ **Grid.ColumnDefinitions[0].Width = 8px**
> 또는
> ✅ **Indent 계산용 Border / Presenter Width**

이게 **진짜 indent** 입니다.

그래서 지금 상황이 이렇게 되는 겁니다:

| 시도                       | 결과 | 이유                   |
| ------------------------ | -- | -------------------- |
| ItemsPresenter Margin 변경 | ❌  | indent 아님            |
| StaticResource override  | ❌  | 참조 안 함               |
| Template 전체 override     | ❌  | indent column이 여전히 8 |
| Padding                  | ❌  | 콘텐츠 내부 여백            |

# ❌ 왜 ItemsPresenter Margin은 절대 안 먹나?

TreeView의 계층 구조는:

```text
부모 TreeViewItem
 └─ ItemsPresenter (자식 TreeViewItem들)
     └─ 각 자식 TreeViewItem 내부에서 다시 Grid indent
```

즉,

* ItemsPresenter는 **“자식 묶음”**
* 실제 들여쓰기는 **각 TreeViewItem이 자기 자신을 그릴 때 결정**

# 🔍 실제 내부 구조 (핵심)

WPF `TreeViewItem`의 표준/VSCode 스타일 템플릿은 **항상 같은 레이아웃**을 씁니다.

```text
[ Indent Column ][ Expander Column ][ Content Column ]
```

### 중요한 점

* 자식이 **없는 노드**도
* ❌ Expander Column을 제거하지 않음
* ✅ **Visibility=Hidden** 으로만 처리

즉,

| 상태    | Expander           |
| ----- | ------------------ |
| 자식 있음 | ▶ / ▼ 표시           |
| 자식 없음 | **보이지 않지만 공간은 유지** |

그래서 👇 이 현상이 생깁니다.

> ❝ 접기 마크가 없는데도 들여쓰기가 있다 ❞

# 🧠 Live Visual Tree에서 보면 이렇게 보임

자식 없는 노드 클릭해보면:

```text
TreeViewItem
 └─ Grid
     ├─ Column 0 : Indent (6px)
     ├─ Column 1 : ToggleButton (Visibility=Hidden, Width=Auto)
     ├─ Column 2 : ContentPresenter
```

👉 **Column 1이 여전히 공간을 차지**

# ❌ 왜 Collapse 안 했을까? (WPF 설계 이유)

Microsoft가 이렇게 만든 이유는:

1. **모든 레벨의 텍스트 정렬 유지**
2. 트리 구조 시각적 일관성
3. 런타임에 자식이 생겨도 레이아웃 재계산 최소화

그래서 기본 동작은:

```xml
Visibility="Hidden"   <!-- 공간 유지 -->
```

❌ `Collapsed` 아님

---

# ✅ 해결 방법 3가지 (원하면 바꿀 수 있음)

---

## ✅ 방법 1 (가장 깔끔): 자식 없으면 Expander Column Width = 0

```xml
<Trigger Property="HasItems" Value="False">
    <Setter TargetName="ExpanderColumn" Property="Width" Value="0"/>
</Trigger>
```

전제: ColumnDefinition에 이름이 있어야 함

```xml
<ColumnDefinition x:Name="ExpanderColumn" Width="Auto"/>
```

✔️ 접기 마크 없는 노드는 indent 제거
✔️ 정렬은 약간 달라질 수 있음

---

## ✅ 방법 2: ToggleButton를 `Collapsed` 처리

```xml
<Trigger Property="HasItems" Value="False">
    <Setter TargetName="Expander" Property="Visibility" Value="Collapsed"/>
</Trigger>
```

⚠️ 단점:

* Column이 `Auto`면 줄어들 수 있음
* 스타일에 따라 효과 다름

---

## ✅ 방법 3 (VSCode/SNMPc 스타일 유지용 추천)

> **Indent Column은 유지하고, Expander만 숨긴다**
> → 지금 보이는 상태 (당신 스샷) 그대로

이게 사실 **SNMPc / VSCode / Windows Explorer** 전부 동일한 동작입니다.

# 🎯 왜 SNMP MIB 트리에서는 이게 특히 잘 보이나?

MIB 트리는:

* Leaf 노드가 매우 많음
* OID 길이가 길어 **정렬 차이 눈에 띔**
* 계층이 깊음

그래서:

> “왜 여긴 ▶도 없는데 밀려있지?”
> 라는 느낌이 강하게 듦

하지만 **계층 정렬을 위한 의도된 공간**입니다.
